# 异世界地图设计软件设计案

## 1. 项目目标

本项目目标是制作一个用于异世界作品参考的地图设计软件，重点服务于以下场景：

- 作者进行世界观地理设计
- 设定师为国家、地区、聚落、生态带提供地图底稿
- AI agent 通过结构化指令修改地图内容
- 用户在 WebUI 中直观看到六角格地图的变化

当前阶段只做设计，不进入实现。

## 2. 已明确需求

根据当前需求，系统至少应满足以下能力：

1. 提供 WebUI
2. 地图以六角形单元格显示
3. 每个单元格具有行列坐标和自己的编号
4. 支持通过指令修改单元格内容，方便 AI agent 调用
5. 单元格内容以“地形 - 生态”叠加方式展示

## 3. 产品定位

建议将本工具定位为“地图设定编辑器”，而不是“游戏运行地图”。

这意味着设计重点应放在：

- 可编辑性
- 可读性
- 指令可控性
- 数据结构稳定性
- 适合设定创作与 AI 协作

暂不优先考虑：

- 实时多人协作
- 高级图形渲染
- 复杂地形生成算法
- 战棋/游戏逻辑系统

## 4. 核心对象模型

### 4.1 地图

地图可以理解为一个六角格二维网格，包含：

- 地图元信息
- 网格尺寸
- 单元格集合
- 图层/标注信息
- 修改历史

建议的地图元信息：

- 地图名称
- 世界/大陆/区域名称
- 地图描述
- 创建时间
- 最后修改时间
- 版本号

### 4.2 单元格

每个六角格单元至少包含：

- 稳定唯一编号
- 行号
- 列号
- 设计状态
- 地形类型
- 生态类型
- 可选标签
- 可选备注

建议的逻辑结构如下：

```text
Cell
- id: string
- row: number
- col: number
- status: "designed" | "undesigned"
- terrain: string | null
- biome: string | null
- tags: string[]
- note: string
```

补充规则：

- `status = designed` 时，`terrain` 必填，`biome` 可为空
- `status = undesigned` 时，`terrain = null`，`biome = null`
- 单元格允许从 `designed` 清空回 `undesigned`
- “边缘一圈未设计格”是系统自动维护出的结果，不单独存为第三种状态

## 5. 六角格坐标与编号设计

### 5.1 为什么需要同时保留“行列”和“编号”

因为使用中会有两类需求：

- 人类创作者更容易理解“第几行第几列”
- AI agent 或程序接口更适合使用稳定且不会因扩图而重排的 `cell_id`

因此建议每个单元格同时保留：

- 行列坐标：`row` + `col`
- 唯一编号：`cell_id`

### 5.2 正式编号与坐标方案

建议采用以下双重标识：

- 显示坐标：`R0C0`、`R-2C5`、`R3C-1`
- 内部编号：`cell@0,0`、`cell@-2,5`

优点：

- 规则直观
- 人眼可读
- 便于指令解析
- 与数据存储一致
- 向上、下、左、右扩展时不需要整体重排行列

补充规则：

- 地图使用固定原点 `R0C0`
- 行列坐标允许出现负数
- 对界面和结构化指令，坐标本身就是最主要的定位编号
- 如果界面空间有限，可对 `R-2C5` 进行缩写显示，但悬停和详情面板必须展示完整坐标

### 5.3 六角格布局方式

已确定采用：

- 平顶六角形（flat-top）
- 偶数列偏移（even-q offset）

这将作为当前版本的基础坐标与渲染规则。

### 5.4 地图扩展方式

地图不采用“固定尺寸画布优先”的方式，而采用“围绕已设计区域逐步扩展”的方式。

正式规则如下：

- 系统只需要存储已出现的单元格
- 已设计单元格外侧始终自动维护一圈 `undesigned` 单元格
- 当边缘的 `undesigned` 单元格被编辑为 `designed` 后，系统继续在更外层补出新的一圈 `undesigned`
- 如果某个内部单元格被清空回 `undesigned`，系统重新计算哪些格子应被保留、哪些格子属于外围冗余空白

这样做的目标：

- 地图可以无限向四周扩展
- 用户不会一开始面对过大的空白画布
- AI agent 可按边界逐步扩图
- 数据存储更适合稀疏地图结构

## 6. 地形 - 生态叠加设计

### 6.1 设计目标

单元格不是只表示一种地表，而是分成两个维度：

- 地形（Terrain）
- 生态（Biome）

这样更适合异世界设定，因为：

- 同样是平原，可以有草原、冻原、湿地草甸等不同生态表现
- 同样是山地，可以是针叶林山地、高山带、冰川山地等不同生态表现

### 6.2 状态与空白规则

单元格状态正式采用两种：

- `designed`
- `undesigned`

具体规则：

- `designed` 表示该格已经有明确地图定义
- `undesigned` 表示该格存在于当前地图边缘，但尚未被正式设计
- 清空一个已设计格时，该格回到 `undesigned`
- `undesigned` 不是地形或生态枚举项，而是单元格状态

### 6.5 展示建议

WebUI 中每个单元格可以采用叠加显示：

- 主层：地形决定基础颜色或底纹
- 副层：生态决定角标、纹理、边框、图案或次级色

例如：

- 平原 + 草原
- 山地 + 针叶林
- 河口 + 咸淡水生态
- 沙丘地 + 荒漠生态

### 6.6 Feature / Tag 补充层与 V1 数据模型

以下项目不直接放入 `terrain` 或 `biome`，而作为附加特征、标签或后续独立图层：

- `island`
- `archipelago`
- `peninsula`
- `bay`
- `strait`
- `isthmus`
- `cape`
- `mountain_range`
- `peak`
- `waterfall`
- `oasis`
- `cliff`
- `crater`
- `cave_entrance`
- `cavern_system`
- `fault_line`
- `shoreline`
- `watershed`
- `ridge`

其中，V1 正式收录以下 12 个预设 tag：

| 中文名 | 内部值 | 显示层级 | 说明 |
| --- | --- | --- | --- |
| 岛屿 | `island` | 详情 tag | 单格表现为岛屿属性 |
| 半岛 | `peninsula` | 详情 tag | 单格表现为半岛属性 |
| 海湾 | `bay` | 详情 tag | 单格位于海湾关键部位 |
| 海峡 | `strait` | 详情 tag | 单格位于海峡关键部位 |
| 海角 | `cape` | 详情 tag | 单格表现为突出的海角 |
| 山峰 | `peak` | 主角标 tag | 需要在近景和导出图中优先识别 |
| 悬崖 | `cliff` | 详情 tag | 表现为陡坎、崖壁、断崖 |
| 瀑布 | `waterfall` | 主角标 tag | 适合在近景和导出图中显示角标 |
| 绿洲 | `oasis` | 主角标 tag | 干旱区的重要标记点 |
| 陨坑 | `crater` | 主角标 tag | 撞击坑或大型环形坑地 |
| 洞穴入口 | `cave_entrance` | 主角标 tag | 洞穴、地下通道入口 |
| 断层线 | `fault_line` | 详情 tag | 地质断裂或活动断层标记 |

以下候选 tag 暂不进入 V1：

- `archipelago`
- `isthmus`
- `mountain_range`
- `cavern_system`
- `shoreline`
- `watershed`
- `ridge`

这些项目更像跨格结构、区域结构或关系结构，适合在后续更完整的图层系统中再引入。

V1 先采用最小数据模型：

- `tags` 在文件中保存为 `string[]`
- 每个 tag 使用稳定英文内部值，界面可映射为中文显示名
- `tags` 的值来自系统内置预设字典，而不是任意自由文本
- 一个单元格允许有 0 到多个 tag
- V1 不为 tag 单独保存强度、方向、跨格范围或附加参数
- V1 不把 tag 做成独立可编辑图层，而是先作为单元格附加层
- V1 不开放用户自定义 `tag` 字典；如需自由补充说明，统一写入 `note`

V1 显示规则正式定为：

- 主角标 tag 只有 `peak`、`waterfall`、`oasis`、`cave_entrance`、`crater`
- 详情 tag 默认不在格内显示角标，只在悬停、右侧详情和导出元数据中完整可见
- 同一单元格存在多个主角标 tag 时，V1 只显示 1 个角标
- 主角标显示优先级固定为：`waterfall > oasis > peak > cave_entrance > crater`
- 无论格内是否显示角标，单元格的完整 `tags` 列表都必须保留在数据中

WebUI 展示建议：

- 右侧详情面板用 tag 列表或 tag chip 作为主要编辑方式
- 地图格内默认不强制展示全部 tag，避免与坐标、terrain、biome 争抢空间
- 在高缩放级别下，可只对少量高价值 tag 显示角标或小图示
- 未在格内直接显示的 tag，仍应在悬停信息和右侧详情中完整可见

### 6.7 显示密度

需要决定单元格内显示多少信息：

方案 A：只显示颜色和图案，详细信息靠悬停查看

方案 B：格内直接显示简写，例如 `PLN-GRS`

方案 C：默认简洁显示，选中后展示完整信息

已确定采用方案 C，并补充为：

- 单元格内默认显示坐标编号，例如 `R-2C5`
- 单元格内默认显示地形 - 生态简写，例如 `MTN-ALN`
- 如果格内空间不足，可根据缩放级别隐藏简写，但坐标优先保留

### 6.8 Terrain / Biome 组合校验规则

V1 不需要一开始就维护完整的“全量兼容矩阵”，但必须定义最小校验规则，避免出现明显违和或自相矛盾的组合。

建议将组合校验结果分成 3 级：

- `valid`：合法，允许直接保存
- `warning`：可保存，但 WebUI 和 CLI 应给出提醒
- `invalid`：不合法，拒绝保存

基础规则：

- 只要 `terrain` 已设置，`biome = null` 始终合法
- 所有硬性校验都以内部值为准，而不是中文显示名
- V1 先采用“少量硬性禁止 + 少量弱警告”的策略，不做庞大组合表

建议直接判为 `invalid` 的组合：

- `freshwater` 不可与 `ocean`、`sea`、`reef` 组合
- `marine` 不可与明显内陆干地地形组合，例如 `plain`、`hill`、`mountain`、`plateau`、`dune`、`badlands`
- `coral` 只允许与 `sea`、`reef`、`lagoon` 组合
- `seagrass` 只允许与 `sea`、`coast`、`lagoon`、`estuary` 组合
- `mangrove` 只允许与 `coast`、`tidal_flat`、`lagoon`、`estuary`、`delta` 组合
- `pack_ice` 只允许与 `ocean`、`sea`、`coast` 组合
- `arid` 不可与稳定水体地形组合，例如 `ocean`、`sea`、`lake`、`river`、`lagoon`

建议判为 `warning` 但允许保存的组合：

- `alpine` 出现在明显低地地形上，例如 `plain`、`alluvial_plain`、`delta`
- `tropical_rainforest` 出现在极寒地形上，例如 `glacier`、`permafrost`
- `bog`、`marsh`、`swamp`、`reedbed` 出现在强起伏高地地形上，例如 `mountain`、`canyon`、`lava_field`
- `bare` 出现在理论上可成立但表现较极端的湿润地形上，例如 `wetland`、`floodplain`

后续如果需要更高精度的现实校验，再补充完整兼容矩阵；V1 先以“避免明显错误、保留创作弹性”为目标。

## 7. 指令驱动修改设计

### 7.1 目标

系统需要支持“通过指令修改地图”，方便：

- AI agent 调用
- 批量编辑
- 自动化生成
- 可回放的编辑记录

### 7.2 指令形式建议

建议支持两类输入形式：

#### 形式一：自然语言近似指令

示例：

- 把第 3 行第 8 列改为山地
- 将 R-2C5 的生态改为针叶林
- 把 R0C2 到 R0C6 设为河流

优点：

- 适合人工输入
- 适合简单 AI 调用

缺点：

- 解析歧义较高

#### 形式二：结构化指令

示例：

```json
{
  "action": "set_cell",
  "target": {
    "row": -2,
    "col": 5
  },
  "changes": {
    "status": "designed",
    "terrain": "mountain",
    "biome": "conifer_forest"
  }
}
```

优点：

- 适合程序和 AI agent 稳定调用
- 易验证
- 易记录历史

缺点：

- 人类直接输入较繁琐

### 7.3 当前建议

已确定系统内部统一使用结构化指令，WebUI 可以额外提供自然语言输入框，再转译为结构化操作。

也就是说：

- 对外可以“像说话一样修改”
- 对内始终转成标准命令

补充规则：

- 结构化指令优先，是 AI agent 的主调用方式
- 结构化指令中的坐标直接使用扩展坐标，可为负数
- 坐标字段统一使用 `row` 和 `col`
- 当前阶段已明确 CLI 的最小契约和推荐子命令集合，但不展开完整的返回码矩阵与所有异常分支

### 7.4 V1 正式支持的基础命令

V1 正式支持以下命令：

- `set_cell`
- `set_cells`
- `clear_cell`
- `replace_terrain`
- `replace_biome`
- `annotate_cell`

批量编辑边界正式定为：

- V1 只支持“显式坐标列表”批量操作，例如 `set_cells`
- V1 支持对现有单元格执行条件替换，例如 `replace_terrain`、`replace_biome`
- V1 不支持刷子涂抹、框选绘制、连通区扩张、洪泛式填充等高自由度批量编辑
- 如果后续需要增强批量能力，优先考虑 `line`、`ring`、`radius` 这类几何型 helper，而不是自由刷子

示例：设置一个单元格

```json
{
  "action": "set_cell",
  "target": {
    "row": -2,
    "col": 5
  },
  "changes": {
    "status": "designed",
    "terrain": "mountain",
    "biome": "conifer_forest"
  }
}
```

示例：清空一个单元格

```json
{
  "action": "clear_cell",
  "target": {
    "row": -2,
    "col": 5
  }
}
```

`clear_cell` 的行为应等价于：

- `status` 改为 `undesigned`
- `terrain` 置空
- `biome` 置空
- 保留 `row`、`col`、`id`

## 8. 最小可行功能范围（MVP 草案）

为了避免项目一开始过重，建议 MVP 只包含：

1. 打开 Web 页面查看六角格地图
2. 从包含原点和一圈 `undesigned` 外环的初始地图开始编辑
3. 单元格显示行列编号
4. 单击单元格查看详细信息
5. 通过右侧详情面板修改单元格
6. 支持修改地形和生态
7. 支持编辑预设 `tags`
8. 支持清空单元格回到 `undesigned`
9. 已设计区域外自动补出一圈 `undesigned` 单元格
10. 变更后立即在地图上刷新显示
11. 支持撤销 / 重做
12. 支持打开、保存、另存为地图文件
13. 支持导入和导出地图 JSON
14. 支持将整张地图导出为图片

MVP 暂不包含：

- 账号系统
- 云同步
- 多人协作
- 大地图分块性能优化
- 高级笔刷
- 自动生成大陆板块

## 9. WebUI 草案

### 9.1 页面分区建议

建议采用“顶栏 + 三栏主体”的布局：

- 顶栏：文件操作 / 视图操作 / 当前地图状态
- 左侧：工具栏 / 图例 / 地形生态面板
- 中间：六角格地图画布
- 右侧：选中单元详情 / 历史记录

### 9.2 基本交互建议

- 点击单元格：选中
- 悬停单元格：显示 `cell_id`、地形、生态
- 在右侧面板修改单元格属性
- 保存后立即刷新地图显示
- 支持撤销 / 重做

### 9.3 WebUI 定位

WebUI 的角色应明确为“人类使用的可视化编辑器”，而不是 AI agent 的主调用入口。

具体约定：

- WebUI 负责渲染、浏览、选中和人工编辑
- AI agent 的自动化调用应通过 CLI 或命令接口完成
- WebUI 与 CLI 最终共用同一套地图数据模型与命令执行规则

### 9.4 顶栏文件操作

WebUI 顶栏建议提供以下操作：

- `新建地图`
- `打开地图`
- `保存`
- `另存为`
- `导入 JSON`
- `导出 JSON`
- `导出图片`

这些操作用于管理项目目录中的地图文件与导出文件，不要求用户手动输入文件路径。

### 9.5 地图选择逻辑

点击 `打开地图` 后，WebUI 应弹出地图列表选择框，列出项目地图目录中的可用地图文件。

每个条目建议显示：

- 地图名称
- 文件名
- 最后修改时间
- 已设计单元格数量

用户点选后加载目标地图。

### 9.6 保存逻辑

建议首版采用“手动保存 + 脏状态提示”：

- 地图发生未保存修改后，顶栏显示“未保存”状态
- 点击 `保存` 时覆盖当前地图文件
- 点击 `另存为` 时以新文件名保存
- 切换地图、导入新文件或关闭页面前，如存在未保存修改，则先提示确认

### 9.7 导入逻辑

建议首版只支持导入原生地图 JSON 文件。

流程如下：

- 用户点击 `导入 JSON`
- 选择本地 JSON 文件
- 系统校验格式是否合法
- 如文件名冲突，提示覆盖或重命名
- 导入成功后将文件写入项目地图目录
- 用户可选择导入后立即打开

### 9.8 图片导出逻辑

图片导出不应依赖当前浏览器视口截图，而应采用“整图离屏渲染导出”。

默认行为建议如下：

- V1 正式支持 `clean` 与 `reference` 两种 PNG 导出预设
- 导出范围支持 `仅已设计区域` 与 `当前活动区域`
- 高级选项中可选择是否包含外围 `undesigned` 外环
- 可选择是否显示坐标
- 可选择是否显示简写
- 可选择是否显示网格线
- 支持选择导出倍率，例如 `1x`、`2x`、`4x`

正式导出范围定义：

- `仅已设计区域`：只按所有 `designed` 单元格计算导出边界
- `当前活动区域`：按当前实际存在的活动单元格集合计算导出边界，包含外围 `undesigned` 外环

范围与 `undesigned` 显示开关的组合规则：

- 当导出范围为 `仅已设计区域` 时，导出边界始终按 `designed` 单元格包围范围计算，与是否显示 `undesigned` 无关
- 当导出范围为 `当前活动区域` 时，导出边界始终按活动区域计算
- 当导出范围为 `当前活动区域` 且“显示 `undesigned` = 是”时，活动区域中的 `undesigned` 单元格参与渲染
- 当导出范围为 `当前活动区域` 且“显示 `undesigned` = 否”时，边界仍按活动区域计算，但自动生成的外围 `undesigned` 外环不参与渲染
- 处于地图内部、且被有意保留的 `undesigned` 单元格仍按空白六角格规则保留显示

首版导出默认值建议如下：

- 导出格式：`PNG`
- 导出倍率：`2x`
- 导出范围：`仅已设计区域`
- 是否包含 `undesigned`：`否`
- 是否显示坐标：`否`
- 是否显示简写：`否`
- 是否显示网格线：`是`
- 网格线样式：弱化显示
- 是否导出交互态：`否`

V1 正式导出预设定义如下：

- `clean`
  默认预设；导出地形底色、生态纹理、网格线和主角标 tag，不显示坐标与地形 - 生态简写，也不导出自动生成的外围 `undesigned` 外环
- `reference`
  用于设定讨论和校对；在 `clean` 基础上额外显示坐标与地形 - 生态简写，仍默认不导出自动生成的外围 `undesigned` 外环

补充规则：

- 处于地图内部、且被有意保留的 `undesigned` 单元格在导出中应保留为空白六角格
- 自动生成的外围 `undesigned` 外环默认不导出
- V1 默认不导出标题和图例，避免版式复杂化

导出时必须移除以下交互痕迹：

- `hover`
- `selected`
- `unsaved edit`
- 临时操作提示
- 右侧面板相关状态

导出图片建议使用统一背景色：

- `#F4F0E6`

导出边界建议自动增加少量留白：

- 默认增加 `32px` 边距，避免地图内容贴边

导出文件默认命名行为：

- PNG 导出默认文件名：`<地图名>-full.png`
- 如文件已存在，应提示覆盖、取消或改名

### 9.9 右侧详情面板结构

右侧详情面板是 WebUI 的主编辑区，建议从上到下固定为以下 5 个区块：

1. 当前选中信息
2. 基础属性编辑
3. 附加信息编辑
4. 操作按钮区
5. 历史摘要区

未选中任何单元格时，右侧面板显示占位提示：

- `请选择一个单元格`

### 9.10 右侧详情面板字段

#### 9.10.1 当前选中信息

该区块只读，不允许直接编辑，建议显示：

- 坐标，例如 `R-2C5`
- 内部 ID，例如 `cell@-2,5`
- 当前状态：`designed` 或 `undesigned`

如果当前单元格属于外围 `undesigned` 边界，可额外显示轻提示：

- `编辑此格会扩展地图边界`

#### 9.10.2 基础属性编辑

该区块是右侧面板的主编辑区域，字段顺序建议固定如下：

- `地形 Terrain`
- `生态 Biome`
- `状态 Status`

其中：

- `Terrain` 使用下拉框编辑
- `Biome` 使用下拉框编辑，允许为空
- `Status` 只读显示，不建议让用户手动切换

状态由系统自动推导：

- 只要用户为一个 `undesigned` 格设置有效 `terrain` 并保存，该格就转为 `designed`
- 点击“清空此格”后，该格回到 `undesigned`

#### 9.10.3 附加信息编辑

该区块建议包含：

- `Tags`
- `Note`

建议规则：

- `Tags` 使用预设 tag 多选控件，可表现为多选下拉、checklist 或 chip picker
- `Tags` 只允许选择 V1 预设 tag 字典中的值，不允许直接输入自由文本
- 如需填写不在预设字典中的补充说明，统一写入 `Note`
- `Note` 使用多行文本框

#### 9.10.4 操作按钮区

首版建议只保留 3 个主按钮：

- `保存修改`
- `清空此格`
- `取消编辑`

按钮行为规则：

- `保存修改`：将当前表单内容应用到选中格
- `清空此格`：将该格重置为真正空白状态
- `取消编辑`：丢弃当前未保存的改动，恢复为本次选中时的原始值

“清空此格”应等价于：

- `status = undesigned`
- `terrain = null`
- `biome = null`
- `tags = []`
- `note = ""`

#### 9.10.5 历史摘要区

右侧面板底部建议显示该格最近一次修改摘要，至少包含：

- 最近修改时间
- 最近修改后的主要结果
- 修改来源，例如 `webui`

### 9.11 右侧面板交互规则

为了降低误操作，建议采用“先改表单，再保存”的编辑流程，而不是字段修改即立刻写入地图。

具体规则：

- 选中 `undesigned` 格时，右侧表单载入空白值
- 选中 `designed` 格时，右侧表单载入已有值
- `biome` 为空时，界面显示“未设置”，但状态仍可保持 `designed`
- 任意字段发生改动后，右侧面板进入“未保存更改”状态
- 顶栏应同步显示地图存在未保存修改
- 如果用户在当前格存在未保存更改时尝试切换到另一个格，系统应先提示确认

切换确认建议提供：

- 放弃修改并切换
- 取消切换

### 9.12 地图缩放层级与信息显示

WebUI 建议采用 4 个明确的缩放层级，而不是完全连续的自由信息切换。

#### 9.12.1 Level 1：超远景

适用于观察地图整体轮廓。

建议显示：

- 六角格基本形状
- `terrain` 底色
- 弱化后的 `biome` 纹理
- `designed / undesigned` 状态差异
- 选中格高亮

默认隐藏：

- 坐标
- 简写
- 备注或标签文本

#### 9.12.2 Level 2：远景

适用于观察地区分布结构。

建议显示：

- 六角格形状
- `terrain` 底色
- `biome` 纹理
- `undesigned` 外环
- 悬停和选中高亮

默认行为：

- 坐标可选显示，但默认关闭
- 地形 - 生态简写默认隐藏

#### 9.12.3 Level 3：中景

这是推荐的默认编辑视角。

建议显示：

- 六角格形状
- `terrain` 底色
- `biome` 纹理
- 坐标
- 悬停和选中高亮

规则：

- `designed` 格默认显示坐标
- `undesigned` 格也显示坐标
- 地形 - 生态简写默认隐藏

#### 9.12.4 Level 4：近景

适用于逐格查看和精细编辑。

建议显示：

- 六角格形状
- `terrain` 底色
- `biome` 纹理
- 坐标
- 地形 - 生态简写
- 更详细的悬停信息

#### 9.12.5 统一显示规则

所有缩放层级共同遵循以下规则：

- `undesigned` 格永远不显示地形 - 生态简写
- 如果格内空间不足，优先保留坐标，隐藏简写
- 如果连坐标也放不下，则只保留颜色、纹理和交互高亮
- 被选中的格可临时增强显示，即使当前层级默认不显示完整文字

### 9.13 WebUI 视觉系统总原则

首版 WebUI 建议采用“低饱和地图图册风”的程序化视觉风格。

正式规则如下：

- `terrain` 决定单元格底色
- `biome` 决定单元格叠加纹理和次级色
- `status` 决定透明度、边框与是否显示简写
- 文字信息始终位于最上层
- 视觉系统不依赖外部美术资源，首版可完全使用程序化绘制实现

### 9.15 Biome 纹理规则

`Biome` 建议通过程序化纹理表达，不直接覆盖 `terrain` 底色。

#### 9.15.1 植被类

- `grassland`: 细点纹
- `steppe`: 稀疏短横线
- `savanna`: 点纹加少量空心小圆点
- `deciduous_forest`: 圆形树冠簇状纹理
- `mixed_forest`: 圆形树冠中混少量三角纹
- `conifer_forest`: 小三角或短竖针叶纹
- `temperate_rainforest`: 高密度深色森林纹
- `tropical_rainforest`: 更密的团簇叶冠纹
- `monsoon_forest`: 中高密度叶冠纹
- `cloud_forest`: 森林纹叠轻雾纹

#### 9.15.2 干旱与灌丛类

- `shrubland`: 小逗点或短弯笔触
- `mediterranean`: 整齐短斜线灌丛纹
- `xeric_shrubland`: 更稀疏的干旱灌丛纹
- `arid`: 细沙纹或浅对角波纹
- `semi_arid`: 更轻的沙纹加少量稀疏点

#### 9.15.3 高寒类

- `tundra`: 稀疏冷色点纹
- `alpine`: 断裂短线加碎点
- `polar`: 极淡冰晶裂纹

#### 9.15.4 湿地与水域生态类

- `marsh`: 湿点和小水斑纹
- `swamp`: 不规则深色湿斑与弯曲细线
- `bog`: 密集细颗粒与闷重软斑块
- `reedbed`: 细长竖线
- `mangrove`: 短分叉根系纹
- `freshwater`: 细短波纹
- `marine`: 舒展长波纹
- `brackish`: 波纹与点纹混合
- `coral`: 小环状或蜂窝状纹理
- `seagrass`: 弧形叶带纹
- `pack_ice`: 碎裂片状线纹

#### 9.15.5 裸地类

- `bare`: 极轻颗粒噪点，尽量保持接近底色

### 9.16 状态与交互样式

单元格除了地形和生态之外，还应有独立的状态与交互层。

#### 9.16.1 状态样式

- `undesigned`: 使用统一留白底色 `#E8E3D9`，搭配 `#B8B1A4` 虚线边框，并叠加极淡斜线底纹
- `designed`: 按 `terrain + biome` 正常渲染，使用实线边框

#### 9.16.2 交互样式

- `selected`: 外描边使用 `#E56A2E`，线宽明显增加，并可叠一层淡白内描边
- `hover`: 边框使用 `#243746`，做轻量高亮
- `unsaved edit`: 可使用 `#C58A1E` 的临时细描边表示当前格已在表单中改动但尚未保存

#### 9.16.3 文字规则

- 第一行显示坐标，例如 `R-2C5`
- 第二行显示地形 - 生态简写，例如 `MTN-CFR`
- `undesigned` 格不显示第二行简写
- 深色底上使用浅色文字
- 浅色底上使用深色文字
- 文字可添加极轻描边以提高复杂底纹上的可读性

### 9.17 地图自动扩展与回收规则

为了让地图能够围绕已设计区域自然生长，同时避免无意义的外围空白无限积累，系统应采用明确的“活动单元格集合”规则。

#### 9.17.1 集合定义

定义如下：

- `D`：所有 `designed` 单元格的集合
- `A`：当前需要在地图上显示、并写入存储文件的活动单元格集合

邻接关系统一按当前已确定的六角格规则计算：

- 平顶六角形
- `even-q offset`

#### 9.17.2 一般规则

当 `D` 非空时，活动集合按以下规则计算：

- `A = D + 所有与 D 相邻的一圈单元格`

这意味着：

- `A` 中属于 `D` 的单元格状态为 `designed`
- `A` 中不属于 `D` 但与 `D` 相邻的单元格状态为 `undesigned`
- 不需要额外引入第三种边界状态

#### 9.17.3 新地图的初始种子区

当地图还没有任何 `designed` 单元格时，系统需要提供一个最小可编辑起点。

建议规则：

- 新建地图时初始化 7 个单元格
- 包含原点 `R0C0`
- 以及原点的 6 个相邻格
- 这 7 个格全部设为 `undesigned`

这组单元格作为“初始编辑种子区”使用。

#### 9.17.4 重算时机

地图的扩展与回收不应在表单字段变化过程中实时抖动，而应在操作提交后统一重算。

建议规则：

- 用户点击 `保存修改` 后统一重算一次
- 用户点击 `清空此格` 后统一重算一次
- 批量命令执行完成后统一重算一次

#### 9.17.5 外围扩展规则

当一个边缘 `undesigned` 单元格被保存为 `designed` 时：

- 该格进入 `D`
- 系统重新计算 `A`
- 新的一圈外围 `undesigned` 自动补出
- 视图不强制跳转

可选的轻提示行为：

- 在界面角落短暂显示 `地图边界已扩展`

#### 9.17.6 冗余空白回收规则

任何 `undesigned` 单元格，只要它不再与任何 `designed` 单元格相邻，就不应继续保留在活动集合中。

具体规则：

- 不再与任何 `designed` 格相邻的 `undesigned` 单元格，从 `A` 中移除
- 被移除的单元格不再显示在画布上
- 被移除的单元格不再写入地图文件

#### 9.17.7 内部空白保留规则

如果某个 `undesigned` 单元格位于已设计区域内部，但仍与至少一个 `designed` 单元格相邻，则：

- 该格保留
- 不进行回收

这允许用户在地图内部保留有意的空白格或重设计区域。

#### 9.17.8 多块独立区域

地图应支持多个彼此不连通的已设计区域同时存在。

规则如下：

- 每个独立 `designed` 区域都各自拥有自己的外围 `undesigned` 外环
- 最终活动集合 `A` 为所有区域活动单元格的并集
- 如果两个区域的外围外环重叠，则直接合并

#### 9.17.9 全部清空时的回退规则

当最后一个 `designed` 单元格被清空后：

- `D` 变为空集
- 地图不保留旧的外围空白形状
- 系统直接回退为“初始编辑种子区”

也就是恢复为：

- 原点 `R0C0`
- 以及原点的 6 个相邻格
- 共 7 个 `undesigned` 单元格

#### 9.17.10 选中状态的处理

重算活动集合后，选中状态应按以下规则更新：

- 如果当前选中的单元格仍存在于新集合中，则保持选中
- 如果当前选中的单元格已被回收移除，则取消选中
- 右侧详情面板恢复为未选中占位状态

### 9.18 地图管理操作

除了打开、保存、导入、导出之外，WebUI 还需要有最小地图管理能力。

V1 建议支持以下操作：

- 新建地图
- 打开地图
- 另存为
- 重命名地图
- 复制地图
- 删除地图

规则建议：

- 新建地图时，立即创建“原点 + 一圈外围 `undesigned`”的初始种子区
- 地图列表以 `meta.name` 为主显示名，同时可显示实际文件名作为副信息
- “重命名地图”优先修改 `meta.name`，不强制立即改动当前文件名
- 如果用户希望文件名同步变化，使用“另存为”完成
- “复制地图”应生成新的 `meta.id`，并将 `revision` 重置为 `1`
- “删除地图”必须进行二次确认，确认文案中应包含地图名
- 如果当前地图存在未保存修改，打开、复制、删除、切换地图前都应先提示确认

### 9.19 错误处理与空状态

V1 虽然不追求复杂流程，但需要把几种高频状态定义清楚，避免界面行为模糊。

建议至少覆盖以下情况：

- 项目中还没有任何地图文件
- 当前地图没有任何 `designed` 单元格
- 未选中任何单元格
- 导入文件校验失败
- 保存失败
- 图片导出失败

界面建议：

- 当 `storage/maps` 为空时，主区域显示空状态，并提供“新建地图”和“导入地图”入口
- 当地图尚无任何 `designed` 单元格时，画布仍显示初始种子区，并提示“从外围空白格开始设计”
- 未选中单元格时，右侧面板保持只读占位态，不显示编辑表单
- 导入失败时，应列出至少一条明确错误原因，而不是只提示“导入失败”
- 保存成功与导出成功应给出轻提示；失败则给出明确原因和失败对象
- 当地图尚无任何 `designed` 单元格时，JSON 保存允许执行，但图片导出默认禁用

## 10. 数据存储草案

建议使用 JSON 作为首个版本的数据格式。

存储策略建议采用“稀疏坐标存储”：

- 只存当前已出现的单元格
- 不强制存储无限大的完整二维数组
- 由程序根据已设计区域和扩展规则维护外围 `undesigned` 单元格

V1 进一步收敛为：

- 地图文件只持久化 `designed` 单元格
- 自动生成的 `undesigned` 外环不写入文件
- 运行时再根据已持久化的 `designed` 单元格推导当前活动区域

### 10.1 原生地图格式

首版正式采用 JSON 作为地图原生格式。

原因：

- 适合保存稀疏坐标结构
- 适合保存 `terrain`、`biome`、`tags`、`note`
- 适合后续扩展更多图层或导出参数
- 比 CSV 更稳定，适合作为长期项目文件格式

CSV 在后续可作为可选的“表格导出格式”，但不作为首版原生保存格式。

### 10.1.1 顶层结构

V1 地图文件正式采用以下顶层字段：

- `schema_version`
- `meta`
- `grid`
- `cells`

示意结构：

```json
{
  "schema_version": 1,
  "meta": {},
  "grid": {},
  "cells": []
}
```

### 10.1.2 `schema_version`

用于表示地图文件格式版本，而不是地图内容修订号。

规则：

- 必填
- 整数
- V1 固定为 `1`

### 10.1.3 `meta`

`meta` 用于保存地图本身的元信息。

建议字段：

- `id`
- `name`
- `world_name`
- `region_name`
- `description`
- `revision`
- `created_at`
- `updated_at`
- `tags`

约束建议：

- `id`：必填，字符串，作为地图稳定 ID
- `name`：必填，字符串，作为地图显示名称
- `world_name`：可选，字符串
- `region_name`：可选，字符串
- `description`：可选，字符串，默认 `""`
- `revision`：必填，整数，初始值为 `1`
- `created_at`：必填，ISO 8601 UTC 时间字符串
- `updated_at`：必填，ISO 8601 UTC 时间字符串
- `tags`：可选，字符串数组，默认 `[]`

补充规则：

- `id` 建议采用 `<slug>-<short_random_suffix>` 格式，例如 `western-grass-sea-7f2k9m`
- `id` 在项目目录 `storage/maps` 内必须唯一
- `id` 一经创建即视为稳定标识，不因重命名地图而改变
- `revision` 在地图首次创建时为 `1`
- 只要发生有效保存，`revision` 就加 `1`
- 如果用户执行保存但地图内容未发生变化，则允许保存成功且不增加 `revision`
- 修改地图元信息，例如 `name`、`description`、`tags`，同样属于有效修改
- `meta.tags` 是地图级标签，用于地图列表分类、搜索和元信息管理，可保持自由字符串

### 10.1.4 `grid`

`grid` 用于保存当前地图采用的六角格规则。

建议字段：

- `layout`
- `origin`

约束建议：

- `layout`：必填，V1 固定为 `flat-top-even-q`
- `origin`：必填，固定为 `{ "row": 0, "col": 0 }`

### 10.1.5 `cells`

`cells` 是地图主体内容，V1 只存储 `designed` 单元格。

这意味着：

- 文件中不写入 `undesigned` 单元格
- 文件中不写入自动生成的外围空白外环
- 文件中不写入运行时 `status`
- 运行时根据 `cells` 重新推导当前活动区域

每个单元格建议字段如下：

- `row`
- `col`
- `terrain`
- `biome`
- `tags`
- `note`

约束建议：

- `row`：必填，整数，允许负数
- `col`：必填，整数，允许负数
- `terrain`：必填，必须在 terrain 字典中
- `biome`：可空；非空时必须在 biome 字典中
- `tags`：可选，字符串数组，默认 `[]`；非空时每个值都必须在 V1 预设 tag 字典中
- `note`：可选，字符串，默认 `""`

补充规则：

- `cells` 中不允许出现重复坐标
- 唯一性由 `row + col` 共同决定
- 建议保存时按 `row asc, col asc` 排序，便于人工阅读与版本对比
- `cells.tags` 是单元格级地理特征标签，必须来自 V1 预设 tag 字典

### 10.1.6 不写入文件的运行时状态

以下信息不进入 V1 地图主文件：

- 自动生成的 `undesigned` 外环
- 运行时 `status`
- `cell_id`
- 当前缩放级别
- 当前选中格
- `hover / selected / unsaved edit` 状态
- 撤销栈与重做栈
- 导出临时选项
- 历史记录明细

推导规则：

- 文件中的 `cells` 一律视为 `designed`
- 运行时内部 ID 由坐标推导，例如 `cell@row,col`
- 运行时再根据 `cells` 自动生成外围 `undesigned` 单元格

### 10.1.7 V1 示例文件

```json
{
  "schema_version": 1,
  "meta": {
    "id": "western-grass-sea-7f2k9m",
    "name": "西境草海",
    "world_name": "阿尔德拉",
    "region_name": "西大陆",
    "description": "草海与山麓交界的设定地图",
    "revision": 3,
    "created_at": "2026-03-25T15:30:00Z",
    "updated_at": "2026-03-25T16:45:00Z",
    "tags": ["草原", "边疆"]
  },
  "grid": {
    "layout": "flat-top-even-q",
    "origin": {
      "row": 0,
      "col": 0
    }
  },
  "cells": [
    {
      "row": -1,
      "col": 1,

    },
    {
      "row": 0,
      "col": 0,
      "terrain": "plain",
      "biome": "grassland",

    }

```

### 10.1.8 加载与运行时恢复规则

加载地图文件后，系统应按以下顺序恢复运行状态：

1. 读取 `cells` 中所有已持久化单元格，并将其视为 `designed`
2. 根据这些单元格重新计算活动区域与外围 `undesigned` 外环
3. 如果 `cells` 为空，则显示“初始编辑种子区”
4. 运行时再生成内部 `cell_id`

### 10.1.9 导入校验规则

导入 JSON 文件时，建议至少执行以下校验：

- `schema_version` 必须存在，且当前支持值为 `1`
- `meta.id` 必须存在，且在项目内不得与现有地图重复
- `meta.name` 必须存在
- `grid.layout` 必须为 `flat-top-even-q`
- `grid.origin` 必须为 `{ "row": 0, "col": 0 }`
- `cells` 必须为数组
- `cells` 内坐标不得重复
- `row` 和 `col` 必须为整数
- `terrain` 必须合法
- `biome` 非空时必须合法
- `tags` 非空时必须全部合法，且每项都必须来自 V1 预设 tag 字典

对于可自动修复的问题，建议导入时补全默认值，例如：

- 缺失 `tags` 时补为 `[]`
- 缺失 `note` 时补为 `""`

对于结构性错误，建议直接拒绝导入，例如：

- 顶层结构缺失
- `meta.id` 缺失或冲突
- 坐标重复
- 非法坐标类型
- 非法 terrain / biome 值
- 非法 tag 值

### 10.2 项目文件目录

建议在项目目录下固定使用以下文件夹：

- `/root/WorkSpace/MapDesigner/storage/maps/`
- `/root/WorkSpace/MapDesigner/storage/exports/`

约定如下：

- `storage/maps` 用于保存地图原生文件
- `storage/exports` 用于保存导出的图片或矢量文件
- WebUI 只管理这两个项目内目录，不提供任意路径浏览

### 10.3 文件命名建议

建议采用以下命名规则：

- 地图文件：`<地图名>.map.json`
- PNG 导出：`<地图名>-full.png`
- SVG 导出：`<地图名>-full.svg`

如果同名文件已存在，应由用户选择覆盖、取消或改名。

V1 地图文件的正式结构以上述 schema 为准。

#### 10.3.1 保存、另存为、复制与重命名规则

V1 正式采用以下文件与版本规则：

- `保存`：覆盖当前地图文件；如地图内容发生变化，则 `revision + 1`
- `另存为`：生成新的地图文件、新的 `meta.id`，并将 `revision` 重置为 `1`
- `复制地图`：生成一份新地图文件，保留内容但生成新的 `meta.id`，并将 `revision` 重置为 `1`
- `重命名地图`：优先修改 `meta.name`，不强制立即改动当前文件名
- 如果用户希望文件名同步变化，使用“另存为”完成
- WebUI 地图列表与 CLI 的 `--map-id` 都以 `meta.id` 作为地图唯一定位键
- 保存时如检测到 `revision` 或文件状态与当前加载版本不一致，则直接报冲突
- V1 不做自动合并

#### 10.3.2 导入时的 `meta.id` 冲突处理

导入地图时，除了检查文件名冲突，还必须检查 `meta.id` 冲突。

规则如下：

- 如果文件名冲突但 `meta.id` 不冲突，可提示覆盖、改名或取消
- 如果 `meta.id` 冲突，V1 不静默覆盖，也不自动改写
- 遇到 `meta.id` 冲突时，建议让用户选择“取消导入”或“生成新 `meta.id` 后导入”
- CLI 模式下如发生 `meta.id` 冲突，默认返回错误；如后续需要自动生成新 ID，应使用显式参数开启，而不是默认行为

### 10.4 导入与导出格式

首版建议支持以下格式：

- 地图导入：`JSON`
- 地图导出：`JSON`
- 图片导出：`PNG`

如果后续地图渲染采用 SVG，可进一步支持：

- 图片/矢量导出：`SVG`

补充约定：

- PNG 导出应为整图渲染结果，而不是浏览器截图
- SVG 导出属于后续增强能力，不要求首版必须完成
- CSV 不作为首版正式导入或项目保存格式

#### 10.4.1 PNG 导出预设

V1 正式提供两个 PNG 导出预设：

- `clean`：适合正式参考图与视觉浏览
- `reference`：适合创作讨论、校对和 AI / 人工共同核对坐标

两种预设都采用以下默认基础参数：

- 背景色：`#F4F0E6`
- 边距：`32px`
- 默认不导出交互态
- 默认不导出自动生成的外围 `undesigned` 外环
- 默认不附加标题和图例

### 10.5 历史记录与主文件边界

V1 不建议将历史记录写入地图主文件。

正式约定：

- `history` 不进入 V1 地图主文件
- 撤销 / 重做作为运行时状态处理
- 如后续需要持久化历史，应单独设计日志文件或快照机制

### 10.6 撤销 / 重做最小规格

V1 建议正式支持撤销 / 重做，但范围只限定在当前打开地图的运行时会话内。

最小规则如下：

- 一次“确认提交”的修改视为一条历史记录
- WebUI 中点击一次 `保存修改` 记为一条历史记录
- 结构化命令中的一次 `set_cell`、`clear_cell` 或一次 `set_cells` 批量操作，各记为一条历史记录
- 导入地图、打开另一张地图、创建新地图前，如当前有未保存改动，应先要求确认
- 成功切换到另一张地图后，当前会话的撤销栈与重做栈清空
- 普通保存不会清空当前撤销栈与重做栈
- V1 建议至少支持最近 `100` 条历史记录

历史项建议至少包含：

- 时间戳
- 操作来源，例如 `webui`、`cli`、`import`
- 操作摘要
- 变更前快照
- 变更后快照

右侧详情面板中的“历史摘要区”只显示适合人类快速查看的摘要，不等价于完整撤销日志。

## 11. AI Agent 接入方向

为了方便 AI agent 后续调用，已确定从一开始就把操作定义成“命令”。

### 11.1 V1 接入边界

V1 正式采用以下分层方式：

- WebUI 只面向人类交互
- AI agent 的主入口是 CLI，而不是通过 WebUI 自动化点击
- WebUI 与 CLI 共用同一套 `map-core`
- `map-core` 负责地图领域规则、坐标规则、校验、扩展回收、命令应用、序列化 / 反序列化规则和历史记录核心逻辑

#### 11.1.1 推荐项目目录结构

建议正式采用以下源码结构：

- `/root/WorkSpace/MapDesigner/apps/web`
- `/root/WorkSpace/MapDesigner/apps/server`
- `/root/WorkSpace/MapDesigner/packages/map-core`
- `/root/WorkSpace/MapDesigner/packages/map-render`

#### 11.1.2 模块职责边界

建议职责划分如下：

- `apps/web`：负责页面布局、交互状态、地图列表、右侧编辑面板与用户提示
- `apps/server`：负责磁盘文件 I/O、本地 API、目录扫描、文件冲突检查、调用 `map-core` 完成导入导出，以及 PNG 生成流程编排
- `packages/map-core`：负责坐标换算、邻接计算、活动区域重算、校验、命令应用、撤销 / 重做核心逻辑、地图数据模型，以及地图数据的序列化 / 反序列化规则
- `packages/map-render`：负责 SVG 六角格布局、terrain 底色、biome 纹理、tag 角标、离屏导出渲染

补充原则：

- `apps/web` 与 `apps/server` 不应各自维护一套独立的地图业务规则
- `map-core` 不直接依赖项目目录、文件路径或具体存储位置
- PNG 导出与 WebUI 地图渲染应尽量共用 `map-render` 的同一套渲染逻辑

### 11.2 V1 命令能力边界

V1 中，CLI 只需要覆盖当前已经确定的地图底图编辑能力：

- 单格设置
- 显式坐标列表批量设置
- 单格清空
- 条件替换
- 单格注记

V1 不要求覆盖以下能力：

- 刷子式连续绘制
- 基于连通区的扩张或涂抹
- 国家、聚落、道路等人文图层编辑
- 关系型河网或跨格路径推导

### 11.3 输入与输出最小契约

V1 建议采用“JSON 输入，JSON 输出”的最小契约。

CLI 工具名建议正式采用：

- `mapdesigner`

推荐的最小子命令集合：

- `mapdesigner maps list`
- `mapdesigner maps create`
- `mapdesigner maps inspect`
- `mapdesigner maps apply`
- `mapdesigner maps import`
- `mapdesigner maps export-json`
- `mapdesigner maps export-png`
- `mapdesigner maps duplicate`
- `mapdesigner maps delete`

输入侧最少支持两种来源：

- `--stdin`
- `--file`

推荐统一使用长参数命名，例如：

- `--map-id`
- `--file`
- `--stdin`
- `--output`
- `--name`

输出侧建议统一返回固定 envelope，至少包含：

- `ok`
- `operation`
- `result`
- `warnings`
- `errors`

其中，`result` 根据命令不同承载对应负载。

示例建议：

- `maps inspect`：`result = { "map": { ... } }`
- `maps list`：`result = { "maps": [ ... ] }`
- `maps apply`：`result = { "map": { ... }, "changed_cells": [ ... ] }`
- `maps delete`：`result = { "deleted_map_id": "..." }`

如 `result` 中包含 `map` 对象，则该对象建议至少包含：

- `id`
- `name`
- `revision`

补充规则：

- 即使执行失败，也尽量返回结构化 JSON，便于 AI agent 稳定解析
- 坐标字段统一使用 `row` 和 `col`
- 结构化命令仍然是唯一标准输入模型；自然语言命令只属于可选的人类友好层

错误对象建议至少包含以下字段：

- `code`
- `message`
- `field`
- `row`
- `col`
- `detail`

其中：

- `field`、`row`、`col` 在不适用时可为空
- `warnings` 建议复用与 `errors` 相同的对象结构

### 11.4 并发与冲突处理

V1 正式采用“单写入者假设”。

这意味着：

- 默认同一时刻只假定有一个入口在修改同一张地图
- 如果保存时发现地图修订号或文件状态与当前加载时不一致，则直接报冲突
- V1 不做自动合并
- 冲突应作为明确错误返回给 CLI 和 WebUI

### 11.5 预留命令类别

可以预留的命令类别：

- `set_cell`
- `set_cells`
- `clear_cell`
- `replace_terrain`
- `replace_biome`
- `annotate_cell`

示例：

```json
{
  "action": "set_cells",
  "targets": [
    {"row": -1, "col": 4},
    {"row": -1, "col": 5},
    {"row": -1, "col": 6}
  ],
  "changes": {
    "status": "designed",
    "terrain": "river",
    "biome": "freshwater"
  }
}
```

## 12. V1 性能目标与验收标准

V1 建议以“中等规模地图可稳定编辑与导出”为性能目标。

建议目标如下：

- 支持约 `2000` 个 `designed` 单元格稳定编辑
- 在上述规模下，活动单元格总量约 `3000` 左右时仍保持可接受交互
- 单格编辑并完成扩展 / 回收重算，目标耗时尽量控制在 `100ms` 量级
- 加载或保存约 `2000` 格地图时，目标耗时尽量控制在 `500ms` 以内
- 导出 `3072px` PNG 时，目标耗时尽量控制在 `3s` 以内

V1 最小验收标准建议如下：

1. 新地图打开后，默认显示“原点 + 6 邻格”的 7 格初始种子区
2. 清空最后一个 `designed` 单元格后，地图正确回退到初始种子区
3. JSON 导入与导出 round-trip 不丢字段、不改语义
4. 非法 `terrain / biome` 组合能稳定返回 `warning` 或 `invalid`
5. 非法 `tags` 值能被导入校验与 CLI 校验稳定识别
6. CLI 对成功、校验失败、保存冲突都能返回稳定 JSON
7. PNG 默认不导出自动生成的外围 `undesigned` 外环
8. 撤销 / 重做至少能稳定处理最近 `100` 条历史记录

## 15. 当前结论

在当前需求下，这个项目最适合被设计为：

“一个面向异世界设定创作、使用平顶六角格、支持扩展坐标与结构化命令操作、以现实向地理底图为核心、由 WebUI 与 CLI 共用统一 `map-core` 的地图编辑器。”

当前 V1 边界已明确为：

- 只处理地理底图，不包含国家、聚落、道路等人文图层
- 使用内置固定的 `terrain / biome / tag` 字典，不开放用户自定义扩展
- 批量编辑仅支持显式坐标列表和条件替换
- AI agent 优先通过 `JSON in / JSON out` 的 CLI 调用
- 项目实现建议采用 `apps/web + apps/server + packages/map-core + packages/map-render` 结构
- 当前设计已经具备开工前所需的主要规格基础

后续所有技术实现都建议围绕这个定位展开。
