# MapDesigner 0.1.0 自部署方案草案

本文档用于明确 `MapDesigner 0.1.0` 的自部署目标、支持范围、交付边界与后续需要补齐的部署交付物。

这不是最终的用户部署手册，而是一份面向发布准备阶段的部署设计说明。

## 1. 目标

`MapDesigner 0.1.0` 的部署目标不是“一键安装所有平台”，而是：

- 让具备基础技术能力的用户能够在自己的机器上完成部署
- 让部署流程清晰、数据目录明确、故障排查有据可依
- 让项目从“开发环境可运行”提升到“发布后可自部署使用”

当前版本的部署重点应放在：

- 本地单机运行
- 本地文件持久化
- WebUI + 本地服务协同工作

## 2. 0.1.0 推荐支持的部署定位

建议将 `0.1.0` 明确定位为：

- 支持源码部署
- 支持本地单机部署
- 支持 Linux / macOS / Windows 上的 Node.js 环境部署
- 不强行承诺 Docker、云部署或桌面安装包

这样做的原因是：

- 符合当前“本地优先”的产品定位
- 与现有技术架构最一致
- 能以最小实现成本提供真实可用的部署能力
- 避免在初始版承诺过大的运维与发行范围

## 3. 建议正式支持的部署模式

### 3.1 模式 A：本地源码部署

这是 `0.1.0` 应正式支持的主模式。

特征如下：

- 用户从源码获取项目
- 本地安装依赖
- 本地启动服务端
- 本地访问 WebUI
- 数据保存在本地目录

这是当前最现实、最稳定、最适合初始版发布的模式。

### 3.2 模式 B：局域网内单机共享访问

这可以作为“原则上可行，但不作为核心承诺”的弱支持模式。

例如：

- 用户在一台机器上部署服务端与前端
- 局域网其他设备通过浏览器访问

此模式理论上可行，但如果 `0.1.0` 尚未系统验证跨设备访问、跨主机路径、端口开放与安全边界，则不建议作为正式主打能力。

## 4. 建议暂不正式支持的部署模式

以下模式建议在 `0.1.0` 中明确标记为暂不作为正式支持范围：

- Docker 镜像部署
- 云服务器公开部署
- 多用户协作部署
- 桌面安装包分发
- 移动端部署
- 托管 SaaS 服务

这些方向不是不能做，而是：

- 需要更多运维、安全、打包和兼容性工作
- 不适合作为 `0.1.0` 的发布重点

## 5. 用户自部署的最小闭环

一个合格的 `0.1.0` 自部署闭环，至少应能让用户完成：

1. 安装运行环境
2. 获取项目源码
3. 安装依赖
4. 启动服务端
5. 启动前端
6. 在浏览器中打开页面
7. 新建地图并保存
8. 在本地目录中找到地图文件和导出文件

只要这条链路稳定清楚，`0.1.0` 就可以视为已经具备初始版自部署能力。

## 6. 当前部署现状判断

基于当前项目状态，可以做如下判断：

### 6.1 已具备的条件

- 项目已有清晰的 monorepo 结构
- 已有 WebUI、Server、CLI 三层职责分离
- 已有本地文件持久化目录
- 已有 `README`、用户说明书、产品总览等文档基础
- 已有 `MAPDESIGNER_ROOT` 环境变量，可覆盖数据根目录
- 已有通过验证的基本运行与测试闭环

### 6.2 尚需收口的部署要点

- 明确用户应该使用开发启动还是生产启动
- 明确前端发布态如何访问
- 明确部署文档中的推荐命令和步骤
- 明确环境变量、目录与备份说明
- 明确常见问题排查方式

## 7. 0.1.0 的关键部署决策

### 7.1 是否允许仅以开发模式作为发布部署方式

不建议仅使用：

- `pnpm dev:server`
- `pnpm dev:web`

作为对外发布的正式部署方式。

原因是：

- 这更像开发者工作流，而不是用户部署工作流
- 用户会混淆“开发调试方式”和“发布运行方式”
- 不利于长期维护发布文档

### 7.2 是否应支持“发布态前端访问”

建议支持。

也就是说，`0.1.0` 最终最好能做到：

- 前端可以构建为发布产物
- 用户可以按发布方式访问前端页面
- 不要求用户长期依赖 Vite 开发服务器

这是当前部署方案里最重要的一项建议。

### 7.3 是否需要数据库

不需要。

当前本地文件存储已经符合 `0.1.0` 的产品目标，部署说明只需把目录、备份和迁移说明写清楚即可。

## 8. 推荐的 0.1.0 部署口径

如果以发布视角描述，建议将 `0.1.0` 的部署能力表述为：

- 支持用户在本地环境中基于源码完成部署
- 支持在 Node.js 20 + pnpm 环境中运行
- 默认使用本地文件进行地图存储和导出
- 适合个人创作、资料整理与本地 AI agent 工作流集成

## 9. 需要补齐的部署交付物

为了让用户真正能够顺利部署，建议补齐以下交付物。

### 9.1 面向用户的部署文档

建议新增：

- `docs/deployment.md`

应覆盖：

- 环境要求
- 安装步骤
- 启动步骤
- 访问地址
- 数据目录
- 常见问题

### 9.2 配置说明文档

建议在部署文档或单独文档中明确：

- 服务端口
- Web 访问地址
- `MAPDESIGNER_ROOT`
- 默认数据目录
- 导出目录

### 9.3 发布运行命令

建议明确并统一以下命令：

- 安装：`pnpm install`
- 构建：`pnpm build`
- 服务端启动：发布态启动命令
- 前端访问：发布态访问方式

### 9.4 FAQ 或排查说明

建议补充常见问题，例如：

- 端口被占用怎么办
- 页面打不开怎么办
- 文件保存到哪里
- 如何备份地图
- 如何迁移数据到另一台机器

## 10. 数据目录与备份建议

部署文档中应明确告诉用户：

- 地图主文件默认位于 `storage/maps`
- 导出文件默认位于 `storage/exports`
- 若需要迁移或备份，应优先备份整个 `storage` 目录

同时应明确：

- 项目升级时，不应随意删除已有 `storage/maps`
- 如果需要更换数据根目录，可使用 `MAPDESIGNER_ROOT`

## 11. 环境变量建议

当前 `0.1.0` 已可明确支持的环境变量重点为：

- `MAPDESIGNER_ROOT`

建议在后续部署文档中明确说明：

- 用途：覆盖默认数据根目录
- 适用场景：多环境部署、测试隔离、自定义存储位置

## 12. 0.1.0 建议的后续实施顺序

如果确定继续推进部署支持，建议按以下顺序执行：

1. 确认是否支持前端发布态访问
2. 明确发布态启动命令
3. 编写部署文档
4. 编写 FAQ / 排查说明
5. 重新核对 README 与用户说明书中的启动和部署部分
6. 在发布前做一次从零部署验收

## 13. 发布前建议验收项

在正式发布 `0.1.0` 前，建议至少完成一次“新用户视角”的部署验收。

建议验收内容：

1. 在干净环境中安装依赖
2. 按部署文档启动项目
3. 成功打开 WebUI
4. 新建一张地图并保存
5. 导出 PNG
6. 验证地图文件与导出文件落盘位置
7. 验证 CLI 可正常读取和修改该地图

## 14. 当前结论

`MapDesigner 0.1.0` 目前已经具备“走向可自部署发布”的基础条件，但要真正把“用户自己部署”作为正式发布能力，需要继续补齐：

- 发布态运行口径
- 部署文档
- 配置说明
- 常见问题
- 一次完整部署验收

因此当前最合理的推进方式是：

- 先把部署目标与支持边界说清楚
- 再补最小必要交付物
- 最后完成一次以用户视角执行的部署验收
