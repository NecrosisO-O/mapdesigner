# MapDesigner 0.1.0 Docker 支持方案草案

本文档用于明确 `MapDesigner 0.1.0` 是否支持 Docker、自部署时的目标形态、最小实现范围与建议交付物。

这是一份方案文档，不是最终使用说明。

## 1. 目标

为 `MapDesigner 0.1.0` 提供一个最小、明确、可维护的 Docker 部署方案，使用户在不手动配置本地 Node.js 运行环境的前提下，也能快速启动项目并通过浏览器访问 WebUI。

本方案的目标不是追求复杂编排，而是：

- 降低部署门槛
- 保持与当前发布态架构一致
- 支持本地数据持久化
- 保持部署方式简单可解释

## 2. 当前适合的 Docker 形态

基于当前项目结构，最适合 `0.1.0` 的 Docker 形态是：

- 单容器
- 单进程主服务
- 容器内运行 `server`
- `server` 直接托管前端构建产物
- 地图数据通过 volume 挂载持久化

这意味着 Docker 版的访问体验应当与当前发布态基本一致：

- 容器启动后
- 浏览器访问映射端口
- 直接打开 WebUI

## 3. 为什么适合做单容器方案

当前项目已经具备以下条件：

- 前端可以构建为静态产物
- `server` 已可托管前端构建产物
- API 与前端访问可以共用同一个服务入口
- 数据使用本地文件保存，无数据库依赖

因此没有必要在 `0.1.0` 阶段拆成多容器架构。

## 4. 建议正式支持的 Docker 范围

建议支持：

- 单机 Docker 运行
- 端口映射访问
- 本地 volume 挂载数据目录
- 使用环境变量指定数据根目录

建议不在 `0.1.0` 正式支持范围内承诺：

- 多容器编排
- 反向代理自动配置
- Docker Swarm / Kubernetes
- 云平台镜像发布流程
- 跨架构镜像矩阵

## 5. 用户体验目标

理想使用方式应尽量简单，例如：

1. 构建镜像
2. 运行容器
3. 打开浏览器访问映射端口
4. 开始使用 WebUI

例如，未来目标体验应接近：

```bash
docker build -t mapdesigner:0.1.0 .
docker run -p 3010:3010 -v /host/path:/data -e MAPDESIGNER_ROOT=/data mapdesigner:0.1.0
```

然后浏览器访问：

- `http://localhost:3010`

## 6. 数据持久化策略

Docker 支持中最关键的一点不是“跑起来”，而是“数据不能随着容器删除而丢失”。

因此建议：

- 必须支持通过 volume 挂载保存地图与导出文件
- `MAPDESIGNER_ROOT` 应指向容器内挂载目录

推荐的容器内数据目录可以统一为：

- `/data`

这样：

- 地图文件位于 `/data/storage/maps`
- 导出文件位于 `/data/storage/exports`

## 7. 需要交付的文件

如果正式实施 Docker 支持，建议最小交付物包括：

### 7.1 `Dockerfile`

用于构建项目镜像。

目标职责：

- 安装依赖
- 构建项目
- 在运行阶段启动 `server`

### 7.2 `.dockerignore`

用于减少无关文件进入构建上下文。

建议忽略：

- `node_modules`
- `storage`
- `.git`
- 本地测试产物
- 其他不参与镜像构建的文件

### 7.3 可选 `docker-compose.yml`

是否需要 `docker-compose.yml` 可以视发布目标决定。

我的建议是：

- 不是必需项
- 如果做，也只保留单服务配置
- 重点作为用户示例，而不是复杂编排工具

### 7.4 Docker 使用文档

建议新增：

- `docs/docker.md`

内容包括：

- 如何构建镜像
- 如何运行容器
- 如何映射端口
- 如何挂载数据目录
- 如何使用 `MAPDESIGNER_ROOT`
- 常见问题

## 8. 构建策略建议

### 8.1 推荐使用多阶段构建

建议优先采用多阶段构建，因为它更适合发布：

- 构建阶段负责安装依赖与编译
- 运行阶段只保留必要产物

优点：

- 镜像更干净
- 运行阶段更轻
- 更适合长期维护

### 8.2 不建议使用极简但混乱的单阶段构建

虽然单阶段构建更快写出来，但问题通常包括：

- 镜像更大
- 包含过多构建期依赖
- 后期不利于维护

因此如果要做，建议直接做一个整洁的多阶段版本。

## 9. 关键技术风险

### 9.1 `sharp`

当前 PNG 导出依赖 `sharp`，因此 Docker 中最需要关注的是：

- 镜像构建时依赖能否正常安装
- 运行时导出 PNG 是否正常

这会是 Docker 验收中的重点项目。

### 9.2 前端静态产物路径

需要确认：

- 前端 build 产物在容器中路径清晰
- `server` 托管静态资源时路径正确

### 9.3 数据目录权限

需要确认：

- 容器内对挂载目录有写权限
- 保存地图与导出图片不会因权限问题失败

## 10. 建议验收项

如果实施 Docker 支持，建议至少完成以下验收：

1. `docker build` 成功
2. `docker run` 成功
3. 浏览器访问映射端口可打开 WebUI
4. 新建地图并保存成功
5. 地图文件能写入挂载目录
6. PNG 导出成功
7. 导出文件能写入挂载目录

## 11. 0.1.0 是否必须包含 Docker

结论建议如下：

- 从“可发布性”角度看：不是必须
- 从“易部署性”角度看：明显加分
- 从“工作量”角度看：只要坚持最小单容器方案，就是可控的

因此，Docker 更适合作为：

- `0.1.0` 的可选增强项

而不是：

- 决定 `0.1.0` 能否发布的前置条件

## 12. 当前建议

当前最合理的建议是：

- 如果要做 Docker，严格限制在最小单容器方案
- 不扩展到复杂编排
- 不改变当前项目结构
- 不把 Docker 支持做成一项大工程

## 13. 后续建议实施顺序

如果决定继续推进，建议顺序如下：

1. 编写 `Dockerfile`
2. 编写 `.dockerignore`
3. 补充 Docker 使用文档
4. 构建镜像并完成一次手工验收
5. 再决定是否追加 `docker-compose.yml`
